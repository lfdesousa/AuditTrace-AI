# ADR-014.3: Makefile-Based Development Workflow with Isolated Virtual Environments

**Status**: Accepted
**Date**: 2026-04-09
**Authors**: Luis Filipe de Sousa
**Phase**: Phase 0
**Related**: ADR-014, ADR-014.2

---

## Context

We need a development workflow that:
1. **Isolates dependencies** - No system-wide Python installations
2. **Reproducible** - Same commands work on any machine
3. **Simple** - Minimal learning curve for new contributors
4. **Fast** - Quick test runs during development

### Options Considered

#### Option 1: Tox (Traditional Approach)
```ini
# tox.ini
[tox]
envlist = py312,lint,format

[testenv]
deps = pytest
commands = pytest
```

**Pros**:
- Industry standard for Python packaging
- Handles multiple Python versions
- Isolated environments

**Cons**:
- Complex configuration
- Slower than direct venv
- Julien Danjou recommends against it for simple projects
- Overkill for single-Python-version projects

#### Option 2: Just (Modern Alternative)
```yaml
# justfile
install:
    python3 -m venv .venv
    .venv/bin/pip install -e ".[dev]"

test:
    .venv/bin/pytest -v
```

**Pros**:
- Simpler syntax than Make
- Cross-platform
- Good for scripting

**Cons**:
- Not available on all systems by default
- Requires installing `just` command
- Less familiar to traditional Python developers

#### Option 3: Makefile (Chosen)
```makefile
# Makefile
install:
	python3 -m venv .venv
	.venv/bin/pip install -e ".[dev]"

test:
	.venv/bin/pytest -v
```

**Pros**:
- Available on all Unix systems (Linux, macOS)
- Familiar to most developers
- Simple, explicit, readable
- Julien Danjou's recommended approach
- Works with GitHub Actions natively
- No additional tools required

**Cons**:
- Windows requires WSL or Git Bash
- Make syntax can be tricky for beginners

---

## Decision

We choose **Option 3: Makefile with `.venv`** for the following reasons:

1. **Julien Danjou's recommendation** - In *Python Engineering*, he advocates for simple, explicit virtual environments over Tox for projects with a single Python version requirement.

2. **No system pollution** - All dependencies installed in `.venv/`, never touching system Python.

3. **Reproducible setup** - `make install` works identically on any Unix system.

4. **Fast iteration** - No Tox overhead, direct venv execution.

5. **CI/CD ready** - GitHub Actions uses Make natively.

### Virtual Environment Structure

```
audittrace-server/
├── .venv/                    # Isolated Python environment (gitignored)
│   ├── bin/
│   │   ├── python
│   │   ├── pip
│   │   ├── pytest
│   │   └── uvicorn
│   └── lib/python3.12/
├── Makefile                  # Development commands
├── .python-version           # Python version (3.12)
├── pyproject.toml            # Project metadata
└── requirements.txt          # Pinned dependencies
```

### Makefile Commands

**Essential Commands**:
- `make install` - Create venv, install all dependencies
- `make test` - Run all tests with 90% coverage check
- `make lint` - Run ruff linter
- `make format` - Auto-format code
- `make typecheck` - Run mypy
- `make clean` - Remove all build artifacts

**Optional Commands**:
- `make test-watch` - Run tests in watch mode
- `make test-unit` - Run fast unit tests
- `make docker-build` - Build Docker image
- `make docker-run` - Run Docker container

## Implementation

### Makefile

```makefile
.PHONY: help venv install lint test clean

help: ## Show this help message
	@echo 'Usage: make <target>'

install: venv ## Create virtual environment and install dependencies
	@python3 -m venv .venv
	@.venv/bin/pip install -e ".[dev]"

test: ## Run all tests with coverage
	@.venv/bin/pytest -v --cov=src --cov-fail-under=90

clean: ## Clean up build artifacts
	@rm -rf .venv/ build/ dist/
```

### Setup Script

```bash
#!/bin/bash
# scripts/setup.sh

# Create isolated virtual environment
python3 -m venv .venv

# Activate and install
source .venv/bin/activate
pip install -e ".[dev]"
```

## Usage Examples

### Local Development

```bash
# First-time setup
./scripts/setup.sh
# or
make install

# Run tests
make test

# Run tests in watch mode
make test-watch

# Lint and format
make lint
make format

# Type checking
make typecheck

# Clean everything
make clean
```

### CI/CD (GitHub Actions)

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Python
        uses: actions/setup-python@v5
        with: {python-version: '3.12'}
      - name: Install dependencies
        run: make install
      - name: Run tests
        run: make test
```

### Docker Development

```bash
# Build Docker image
make docker-build

# Run in container
make docker-run
```

## Comparison: Tox vs Makefile

| Aspect | Tox | Makefile |
|---|---|---|
| **Setup complexity** | High (tox.ini) | Low (Makefile) |
| **Installation time** | Slower | Faster |
| **Learning curve** | Steeper | Gentle |
| **Multi-Python support** | ✅ Yes | ❌ No (single version) |
| **Cross-platform** | ✅ Yes | ⚠️ Unix only |
| **Julien Danjou recommendation** | ❌ Overkill | ✅ Preferred |
| **CI/CD compatibility** | ✅ Yes | ✅ Yes |

### When to Use Tox

Tox is appropriate when:
- Multiple Python versions (3.9, 3.10, 3.11, 3.12)
- Complex dependency matrices
- Testing across different environments

**Our case**: Single Python 3.12 → **Makefile is better**

## Best Practices

### 1. Always Use `.venv/` (Not `venv/`)

```bash
# ✅ Correct
python3 -m venv .venv
source .venv/bin/activate

# ❌ Wrong (can conflict with system venv)
python3 -m venv venv
```

### 2. Never Install System-Wide

```bash
# ✅ Correct
make install  # Installs in .venv

# ❌ Wrong
pip install pytest  # Installs system-wide (DON'T DO THIS!)
```

### 3. Pin Python Version

`.python-version`:
```
3.12
```

Ensures all developers use the same Python version.

### 4. Use `.venv/bin/` Explicitly

```bash
# ✅ Correct
.venv/bin/pytest tests/
.venv/bin/ruff check src/

# ❌ Wrong (depends on activation)
pytest tests/
ruff check src/
```

### 5. Clean Up Regularly

```bash
# Monthly cleanup
make clean
make install
```

## Troubleshooting

### "python3: command not found"

**Solution**: Install Python 3.12
```bash
# macOS
brew install python@3.12

# Ubuntu
apt-get install python3.12
```

### "Permission denied" on `.venv/`

**Solution**: Fix permissions
```bash
chmod -R u+rwx .venv/
```

### "Module not found" after `make install`

**Solution**: Ensure venv is activated
```bash
source .venv/bin/activate
```
