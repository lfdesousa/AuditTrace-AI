# ADR-027: MinIO Object Storage for Stateless Memory Layers

**Status:** Accepted
**Date:** 2026-04-13
**Deciders:** Luis Filipe de Sousa
**Related:** ADR-018 (4-layer memory), ADR-019 (ChromaDB server mode),
ADR-021 (TLS + Traefik), ADR-026 (multi-user identity + RLS)

## Context

Layers 1 (episodic / ADRs) and 2 (procedural / skills) of the 4-layer memory
architecture (ADR-018) were backed by host filesystem bind mounts:

```yaml
volumes:
  - ./memory/episodic:/app/memory/episodic:ro
  - ./memory/procedural:/app/memory/procedural:ro
```

This violated 12-factor principle VI (stateless processes) and created a
security incident where host directory contents were exposed via a
misconfigured mount. The memory-server container must be fully stateless —
no host filesystem coupling for runtime data.

Additionally, the system needed per-user isolation for private knowledge
(research papers, coursework) that follows the same trust boundary as
PostgreSQL RLS (ADR-026 §16) and ChromaDB user_id filtering.

## Decision

Replace filesystem bind mounts with **MinIO** (S3-compatible object storage)
as a 12-factor backing service. MinIO runs as a Docker Compose service alongside
PostgreSQL, ChromaDB, and Redis.

### §1. Bucket layout — two-tier isolation

```
memory-shared/                     ← shared content, all authenticated users
  episodic/
    ADR-014-python-package-structure.md
    ADR-018-four-layer-memory-port.md
    ...
  procedural/
    SKILL-ARCHITECTURE.md
    SKILL-GENAI.md
    ...

memory-private/                    ← per-user content, JWT sub as prefix
  {user_id}/                       ← Keycloak sub claim (e.g. kc-luis-001)
    ai-research/
      main.pdf
      ...
    scm-coursework/
      transcripts/SC0x_M1U2V1-en.txt
      notes/GoNuts Juice Company.md
      ...
```

**Shared content** (ADRs, skills) is organisation-wide knowledge — any
authenticated request can read. **Private content** is user-scoped — the
path prefix is derived from the validated JWT `sub` claim, never from
user input.

### §2. AuthZ model — JWT-driven, application-mediated

The memory-server is the **sole mediator** of all MinIO access. End-user
JWTs never reach MinIO — MinIO sees only the memory-server's service account
credentials. This matches the existing PostgreSQL RLS pattern where the app
calls `set_config('app.current_user_id', sub)` and the database enforces
row filtering.

| Content tier | Bucket | Path scoping | Trust boundary |
|---|---|---|---|
| Shared | `memory-shared` | None — any authenticated user reads all | `require_user` succeeds (valid JWT) |
| Private | `memory-private` | `{user_context.user_id}/` prefix | JWT `sub` claim → path prefix |
| Private (admin) | `memory-private` | No prefix — admin reads all users | `user_context.is_admin == True` |

**Same isolation pattern across all backing services:**

| Service | Shared content | Per-user isolation mechanism |
|---|---|---|
| PostgreSQL | N/A | RLS policy on `app.current_user_id` GUC (ADR-026) |
| ChromaDB | `decisions`, `skills` collections | `where={"user_id": ...}` filter (ADR-026 Phase 2) |
| MinIO | `memory-shared` bucket | `{user_id}/` path prefix on `memory-private` |

### §3. Encryption at rest — SSE-S3

MinIO server-side encryption with auto-generated keys, enabled via:

```yaml
MINIO_KMS_SECRET_KEY: "sovereign-key:${AUDITTRACE_MINIO_KMS_KEY}"
```

The KMS key is generated by `scripts/setup-secrets.sh` (32 bytes, hex-encoded)
and stored in `secrets/minio_kms_key.txt`. The Docker volume (`minio_data`)
holds only ciphertext — a volume breach exposes encrypted blobs, not readable
documents. Zero application code changes; the `minio` Python SDK writes objects
normally and MinIO encrypts transparently.

### §4. Encryption in transit

Internal Docker network traffic follows the ADR-021 pattern: plain HTTP between
containers on `audittrace-net`, TLS terminated at Traefik for external access.
MinIO's S3 API port (9000) is not exposed through Traefik — only the
memory-server communicates with MinIO. The MinIO console (9001) is exposed on
`localhost:19001` for development only.

**Production path (Phase 4 / Istio):** mTLS between all containers including
MinIO, same as PostgreSQL and ChromaDB. No application changes needed.

### §5. Service configuration

```yaml
minio:
  image: minio/minio:latest
  container_name: sovereign-minio
  command: server /data --console-address ":9001"
  environment:
    MINIO_ROOT_USER: ${AUDITTRACE_MINIO_ACCESS_KEY:-minioadmin}
    MINIO_ROOT_PASSWORD: ${AUDITTRACE_MINIO_SECRET_KEY}
    MINIO_KMS_SECRET_KEY: "sovereign-key:${AUDITTRACE_MINIO_KMS_KEY}"
  volumes:
    - minio_data:/data
  healthcheck:
    test: ["CMD-SHELL", "mc ready local || exit 1"]
    interval: 10s
    timeout: 5s
    retries: 5
    start_period: 10s
```

### §6. Application integration

New S3-backed service implementations replace the filesystem-backed ones:

- `S3EpisodicService` reads `memory-shared/episodic/ADR-*.md` → caches in memory
- `S3ProceduralService` reads `memory-shared/procedural/SKILL-*.md` → caches in memory

Both implement the same ABC as `FileEpisodicService` / `FileProceduralService`
(ADR-018). The `File*` implementations remain as fallbacks for local development
and tests without MinIO. Service selection is driven by `AUDITTRACE_MINIO_SECRET_KEY`:
when set, S3 services activate; when empty, filesystem fallback.

### §7. Ingestion tooling

Two scripts handle content lifecycle:

- `scripts/seed-memory.py` — creates buckets, uploads ADRs + skills to
  `memory-shared`, uploads private knowledge to `memory-private/{user_id}/`
- `scripts/index-chromadb.py` — extracts text from all sources (including PDFs
  via `pymupdf`), chunks, and indexes into ChromaDB collections

### §8. Defence in depth (Phase 4)

Bucket policies restricting `memory-private` to path-prefix-scoped operations
are flagged for Phase 4 alongside Istio mTLS. This adds a storage-layer safety
net behind the application-layer prefix enforcement.

## Consequences

### Positive
- Stateless containers — no host filesystem bind mounts for memory data (12-factor VI)
- Per-user isolation consistent with PostgreSQL RLS and ChromaDB patterns
- Encryption at rest — volume breach yields only ciphertext
- Clean separation: shared knowledge vs private knowledge
- `File*Service` fallback preserves zero-infrastructure dev/test experience

### Negative
- New container dependency — MinIO must be running for S3 services to work
- New secrets to manage (`AUDITTRACE_MINIO_*` + KMS key)
- Object listing is slower than filesystem glob (mitigated by in-memory caching)

### Neutral
- `pymupdf` dependency added for PDF text extraction (indexing only)
- `memory/episodic/` and `memory/procedural/` directories remain in repo as seed sources
