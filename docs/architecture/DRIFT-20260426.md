## New items surfaced 2026-04-26

### D9 — `sequence-vault-injection.md` referenced future Istio posture pre-validation

The original `sequence-vault-injection.md` shipped in M1 PR #16
described the in-mesh discipline aspirationally. The first live
install (2026-04-25) deviated from that posture (wholesale Istio
bypass) and was caught by Luis. PR #17 restored the in-mesh
posture; this PR adds the explicit §"Istio mesh posture" subsection
to sequence-vault-injection.md showing the runtime invariants
(scoped exclusion on :8080 only, AuthorizationPolicy on :8200,
TokenReview JWT read live).

**Fix in this PR.** Closed.

### D10 — ADR-043 lacked a Postmortem section for the live install

ADR-045 set the precedent of capturing PM-1..PM-N rules from the
first live install of a new chart-layer feature. ADR-043 was
shipped without that section because the live install hadn't
happened yet. Seven hot-fixes later it has.

**Fix in this PR.** ADR-043 §Postmortem PM-1..PM-7 added. Closed.

### D11 — `keycloak/realm-audittrace.json` does not yet have an `identityProviders` key

This is *expected* state pre-M2; logged here so the M2 drift audit
records that the file's current shape is the baseline ADR-044 will
edit. M2 PR adds the `identityProviders` block + `identityProvider
Mappers` + the four broker types (keycloak-local, google, okta,
entra). Realm import remains idempotent; `Realm 'audittrace' already
exists. Import skipped` on subsequent boots is the documented
behavior (per the Vault live install logs 2026-04-25).

**Closed in M2 PR.** Realm JSON now has explicit empty
`identityProviders[]` and `identityProviderMappers[]` arrays as the
no-federation baseline; per-deployment broker config is added by
`scripts/setup-idp-federation.sh` per ADR-044 §7.

### D12 — `workspace.dsl` has no "Organisational IdP" external actor

The C4 Context view in `workspace.dsl` shows Keycloak as the
identity boundary. ADR-044 adds an external actor "Organisational
IdP" (Entra / Okta / Google Workspace) brokered through Keycloak.

**Closed in M2 PR.** New `organisationalIdP` softwareSystem in
workspace.dsl with the brokerage relationship from Keycloak
(storeToken=false / validateSignature=true / syncMode=FORCE per
ADR-044). DSL validates; the four views pick up the new actor via
`include *`.

### D13 — sequence-oauth2-flow.md does not show the broker hop

The current login sequence ends at Keycloak issuing the JWT. The
M2 path adds an upstream hop: user → external IdP → Keycloak
brokers → JWT.

**Closed in M2 PR.** New §"Federated login via brokered IdP" added
inline to `sequence-oauth2-flow.md` (default per ADR-044's deferred
choice — the diagram remains readable without a separate file).
Covers the JIT-provisioning and returning-user paths, the
syncMode=FORCE refresh-on-every-login behavior, and the post-login
hand-off to the multi-issuer JWT validation path that was already
in place per ADR-032 §2.
