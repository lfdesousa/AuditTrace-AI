# ADR-014.4 — Observability: logging everywhere + OpenTelemetry

- **Status:** Accepted
- **Date:** 2026-04-09
- **Phase:** 0 (enhancement)
- **Supersedes/Extends:** ADR-014.2 (Logging & Dependency Injection)

## Context

ADR-014.2 introduced `@log_call` as a FluentPython-style aspect but only
applied it to the service layer. A review during Phase 0 follow-up
revealed two gaps:

1. **Coverage of the aspect was partial.** Routes, the DB factory layer,
   the DI container, and the concrete `ChromaMemoryService` had no
   logging at all. For a product whose selling point is
   *"reconstructible by design / full audit trail"*, invisible
   request entry/exit and factory calls are a real compliance gap.
2. **Metrics story was missing.** The roadmap mentions Langfuse for
   traces but had no vendor-neutral story for per-operation latency,
   error rates, or request counts. Regulated clients expect
   OpenTelemetry-shaped metrics that can be shipped to any OTLP
   collector (Grafana, Honeycomb, vendor SIEM, etc.).

## Decision

### 1. Logging everywhere, stdout only

`@log_call` is applied to every public function in:

- `routes/{chat,context,audit,session,health}.py`
- `db/factory.py` (all factories, the mock client, the mock collection)
- `dependencies.py` (container + helpers)
- `services/memory.py` (on the concrete `ChromaMemoryService` /
  `MockMemoryService` — **not** the abstract base, where decorators are
  dead because subclasses override)

The decorator supports both sync and async functions. It emits
`INPUT` / `OUTPUT` / `DURATION` records at DEBUG level and `ERROR`
records at ERROR level. All records go to stdout via a single
`StreamHandler(sys.stdout)`. No file handlers exist anywhere in the
codebase — this is asserted by `test_setup_logging_uses_stdout_only`.

Logs also carry the active OTel trace/span IDs automatically via
`LoggingInstrumentor`, so log lines and traces can be correlated by
`otelTraceID` in any structured log backend.

### 2. OpenTelemetry as the single observability aspect

A new module `sovereign_memory.telemetry` initialises a `TracerProvider`
and `MeterProvider` at application startup. `@log_call` is extended so
every decorated call:

- opens a span named after the operation (`ClassName.method`)
- records a histogram sample on `sovereign.operation.duration` (seconds)
- bumps a counter `sovereign.operation.errors` on exception

FastAPI request spans are added independently via
`FastAPIInstrumentor.instrument_app(app)` in the lifespan handler —
one call, zero per-route code.

The aspect gives us **logs + traces + metrics** from a single
decorator. Nothing else in the codebase has to know about OTel.

### 3. Phase 0 runs OTel in no-op mode

When `SOVEREIGN_OTLP_ENDPOINT` is empty (the Phase 0 default), no
exporter is attached. Spans and metrics are still created locally so
`@log_call` works identically in tests and production, but nothing is
shipped off-process. Phase 1 will supply an OTLP collector in the
Docker Compose stack and set the endpoint; no code changes required.

### 4. Tests use the same aspect

`tests/conftest.py` has a session-scoped autouse fixture that calls
`setup_logging(level="DEBUG")` and `telemetry.init_telemetry(...)`
with `otlp_endpoint=""`. This means:

- every test run produces the same structured log lines as production
- `caplog` can assert on `INPUT` / `OUTPUT` / `DURATION` records
- `tests/test_observability.py` exercises both the sync and async
  branches of `@log_call`, the error branch, the bare `@log_call`
  form, the structured formatter, and the telemetry no-op path

## Configuration

New settings in `sovereign_memory.config.Settings` (all `SOVEREIGN_`-prefixed):

| Setting             | Default                       | Purpose                                |
|---------------------|-------------------------------|----------------------------------------|
| `otlp_endpoint`     | `""` (no export)              | OTLP/HTTP collector URL                |
| `otel_service_name` | `sovereign-memory-server`     | `service.name` resource attribute      |
| `tracing_enabled`   | `true`                        | Install tracer provider                |
| `metrics_enabled`   | `true`                        | Install meter provider                 |

New dependencies (pinned in `pyproject.toml` and `requirements.txt`):

```
opentelemetry-api>=1.27.0
opentelemetry-sdk>=1.27.0
opentelemetry-exporter-otlp>=1.27.0
opentelemetry-instrumentation-fastapi>=0.48b0
opentelemetry-instrumentation-logging>=0.48b0
```

## Consequences

**Good:**
- Every request, every factory call, every service call is visible in
  logs, traces, and metrics from a single decorator.
- Vendor-neutral: any OTLP-compatible backend works; no code change to
  switch.
- Tests observe the production logging path — no divergence.
- Phase 0 runs with zero external dependencies (no-op mode), Phase 1
  flips a single environment variable to activate export.
- Coverage enforced at 90% still holds (91.2% at time of writing).

**Trade-offs:**
- Aspect fires on every decorated call; histogram cardinality is bounded
  by operation name, not by request attributes, to keep metric volume
  under control.
- `@log_call` input/output logging is DEBUG-gated; INFO-level production
  runs see only errors and startup lines — no PII leakage by default.

## Pre-existing bugs fixed in the same change set

While wiring the aspect it became clear that Phase 0's test suite had
never actually been runnable. Fixes shipped alongside this ADR:

- `services/memory.py`: `from typing import list, dict` (invalid in
  Python 3.12) → `from typing import Any`
- `pyproject.toml`: spurious `pytest11` entry point registering
  `sovereign_memory.server` as a pytest plugin → removed
- `config.py`: `cors_origins: str = [...]` type/default mismatch →
  `list[str]`; missing `env_prefix="SOVEREIGN_"` → added
- `server.py`: router prefix `"/"` violates FastAPI's no-trailing-slash
  rule → removed
- `db/factory.py`: `MockCollection.add` didn't iterate `ids` and
  returned fabricated identifiers; `MockChromaDBFactory.get_client`
  returned `self` so two calls produced the same object → split into
  factory + `_MockChromaDBClient` wrapper
- `tests/test_dependencies.py`: two tests referenced a local container
  but the functions under test mutate the global → rewritten to check
  the global container
